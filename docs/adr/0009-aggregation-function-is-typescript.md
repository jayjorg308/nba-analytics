# The aggregation function is TypeScript, closing ADR-0007's deferred call

The v1 player-side aggregation (diet-weighted PPS, making deltas, suppression/small-sample flags) lives in **TypeScript** — `src/domain/aggregate.ts`, one pure function over an array of enriched shots, consumed behind the Zod load boundary. This closes the compute-location A/B that ADR-0007 deliberately left open, in favor of its stated lean (B). Python's ownership is unchanged: pull → validate → enrich → roll up → persist the metric-free payload.

**Why:** ADR-0007 set the condition for closing the call — after the pre-build pull — and the pull is done: ADR-0008 fixed the zone set from real counts, so the function became fully specifiable before its engine was committed. The lean's logic holds as stated there: the payload already persists enriched per-shot rows, so the data cost of v2 filters is being paid either way, and betting against them is inconsistent. Siting the function in TS makes v2's filtered re-aggregation a client-side call over subsets — the v1 all-shots call is just the all-pass case.

**Cost accepted:** basketball logic now spans two languages (Python owns enrichment/rollup, TS owns the player-side metrics). Contained by the seam ADR-0007 locked, plus a committed cross-language golden fixture (`tests/fixtures/` — the derive step and the Zod schema must both agree with it, so contract drift breaks a test suite, not production) and a `schemaVersion` literal pinned on both sides.

**Consequences:** the payload stays metric-free and the ADR-0007 guardrails stand unchanged — no headline metrics folded into the persisted payload, no scattering of aggregation across call sites (the function keeps a single production call site, added by the shot-chart increment). As a pure, tested unit, a future port back to Python remains a contained rewrite, not an architecture unwind. ADR-0007 is not superseded: its seam and payload lock remain in force; only its deferred question is now answered.
