# Zone-point conflicts are dropped at derive and counted, never guessed

A raw shot row can contradict itself: `SHOT_TYPE` (the scorer's call — the points the shot was actually worth) versus the coordinate-derived `SHOT_ZONE_BASIC`. Found in the wild: Keyonte George 2025-26 contains a step-back scored **2PT** but zoned **Above the Break 3** with release coordinates ~23.95 ft out — almost certainly a foot on the line. The derive step now **drops** such rows and **counts** them in `_meta.zoneConflictsDropped` (schema v2); the UI reports the count whenever it is nonzero.

**Why:** zone boundaries *are* point-value boundaries at the evaluation grain (the whole PPS model — ADR-0001 — prices a zone at one point value), so a "2PT attempt from a 3PT zone" is unrepresentable, and the payload contract enforces pointValue–zone agreement. The alternatives were both worse. Failing the whole snapshot (the original behavior) lets one bad NBA row veto 880 good ones — it blocked the second hero outright. Reassigning the zone from `SHOT_TYPE` (ATB-3 + 2PT → Mid-Range) fabricates a zone assignment the NBA never made — the same class of invented data ADR-0012 exists to keep out. Exclude-and-report follows the backcourt precedent exactly: removed from evaluation, counted, surfaced — never hidden.

**Consequences:** `_meta.totalShots` is the post-drop count (the schema asserts it equals `shots.length`); dropped rows survive only in the raw layer. The count is a data-quality signal worth glancing at per hero: 1-in-881 is scorer noise, but a materially higher rate in some future snapshot means investigate upstream, not accept silently. The v1→v2 schema bump moved the Zod schema, the derive step, the regenerated golden, and the re-synced deployed payload together, per the cross-language convention (`tests/fixtures/README.md`).
