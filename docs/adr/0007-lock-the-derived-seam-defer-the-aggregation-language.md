# Lock the Python/frontend seam and the derived payload; defer the aggregation language

Python owns the pipeline: pull → validate → enrich → roll up the league baseline (with the ADR-0004 reconciliation test) → persist a typed, Zod-validated JSON payload of `{ enriched per-shot rows + rolled-up zone baseline }`. The v1 player-side aggregation (diet-weighted PPS, making deltas, suppression) is written as **one pure function over an array of enriched shots**, called once with all shots. The **language of that function (Python vs TS) is deliberately left open.**

**Why:** For v1's actual shipped behavior (one player, one season, no filters), computing metrics in Python (A) or in TS (B) renders the identical screen — the only case for siting the engine in TS is that v2 filters re-aggregate over subsets and would be cheap there. That is a bet on unbuilt v2 whose cost is basketball logic split across two languages (a permanent legibility cost). We decline to pay that on faith right before validating v1 on real data. The payload shape is identical either way, so storage is unblocked; the function's exact contents depend on still-unresolved data-dependent calls (mid-range split, corner split, Gate 2), so fixing its language now would commit an engine before its shape is known.

**Consequences:** The derived payload and the seam are fixed and safe to build against. The aggregation stays a single pure, tested unit so a later Python↔TS port is a contained rewrite, not an architecture unwind. If forced, the lean is (B) — the data cost of filters is already being paid, so betting against them is inconsistent — but the call is deferred until after the pre-build pull. A future contributor must not fold the headline metrics into the persisted payload (that would recouple the contract to compute-location) nor scatter the aggregation across multiple call sites (that would break the single-pure-function property).
