# The app consumes a committed deployment copy of the derived payload, synced explicitly

The deployed app reads the derived payload from a **committed copy** at `public/data/<player-slug>/<season>.json`, refreshed only by an explicit sync (`npm run hero:sync` → `scripts/sync-hero-payload.mjs`, which copies the latest `data/derived/<slug>/<season>/*.json`). This completes the storage story as three layers: **raw** (append-only, gitignored — ADR-0006) → **derived** (recomputed from the latest raw, gitignored, overwritable) → **deployed** (committed, explicitly synced). The app fetches this file and Zod-parses it at the load boundary; it **never calls the NBA API**.

**Why:** stats.nba.com is an unofficial endpoint that blocks cloud/data-center IPs — pulls run on a dev machine, never from the deployed app or CI — so the shipped product must carry its own data. Committing the deployed copy (rather than un-ignoring `data/derived`) keeps the working layers regenerable-and-disposable while making deployment reproducible from a clean clone. The explicit sync makes "which data is live" a reviewed, diffable act instead of a build side effect.

**Record correction (near-miss):** `.gitignore`'s unanchored `data/` also matched `public/data/`, which would have silently kept the deployment copy out of git while everything kept working locally. The pattern is now anchored as `/data/` — do not "simplify" it back.

**Consequences:** swapping heroes or seasons means editing `src/heroConfig.ts`, the `hero:sync` arguments in `package.json`, and re-running the sync — drift between config and payload breaks the HeroPage test, and staleness against `data/derived` breaks the (skippable) realPayload test. The deployed copy is the metric-free derived payload verbatim, so committing it does not conflict with ADR-0007's "no metrics in the persisted payload." A future contributor must not point the app at the API or at `data/`, and must not fold the sync into the build.
