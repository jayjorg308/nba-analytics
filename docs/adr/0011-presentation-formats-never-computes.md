# The presentation layer formats metrics; it never computes them

Every number the UI shows comes from the single `aggregateShotMetrics` output (one production call site: `HeroReady` in `src/app/HeroPage.tsx` — ADR-0007/0009). Components format and arrange; they never derive — no rates, shares, PPS values, deltas, or combined groupings are computed in the UI. All rounding lives in `src/format.ts`. The chart stack is **hand-rolled SVG themed by the shared CSS variables — no chart libraries**.

**Why:** ADR-0007/0009 put all basketball math in one pure, tested unit; every ad-hoc calculation in a component is a second implementation that can silently disagree with it. The canonical refusal: there is deliberately no UI-side "combined corners" row, because merging corners would re-derive FG%/PPS outside the function. Rounding scattered through components makes displayed numbers irreproducible in tests. Chart libraries were declined because the honesty details this product turns on — skip-don't-clamp for off-frame dots, muted-never-deleted rows, flag-never-suppress markers, the exact "vs league average" labeling (ADR-0002) — are first-class requirements: owning the SVG makes them straightforward, keeps the runtime dependency-free, and drives both themes from one set of variables.

**Consequences:** when a view needs a number the aggregation does not output, extend `ShotMetrics` and its tests — do not compute it in a component. The line to hold: presentation *mappings* over existing outputs (e.g. a color scale over `makingDelta`, dot placement from `locX/locY`) are legitimate; producing new *quantities* is not. A future contributor must not add a charting dependency to v1 surfaces, introduce a second aggregation call site, or round anywhere but `format.ts`.
