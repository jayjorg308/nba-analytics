# The app ships dark-only

The page renders the dark theme for every visitor, regardless of OS preference: the dark variable set in `src/index.css` / `src/App.css` **is** the theme (`:root`, `color-scheme: dark`), and the light set plus the `@media (prefers-color-scheme: dark)` overrides are removed rather than kept as a dead branch.

**Why:** the dark presentation is the designed look — the B&W poster banner, white ink, and the "Ember" making scale were all tuned on the dark surface, and it is the presentation we want every reader to get. An OS media query cannot be overridden from page CSS, so "always dark" structurally means dark-as-only-theme; keeping an unreachable light palette would leave hexes and a guard branch that can only drift. A `data-theme` toggle mechanism was rejected because nothing toggles it — it would duplicate every hex to buy an option we are deliberately not offering.

**Consequences:** ADR-0014's palette contract now covers one theme: the guard (`src/chart/makingScale.contrast.test.ts`) parses the single `:root` set and asserts the same invariants at the same floors (≥ 4.5:1 label contrast, per-arm monotone luminance lightening outward, gray neutral) — dark-only narrows the guard's scope, never its strength. The dark-mode hue constraint is now global: the 4.5:1 label floor caps warm fills near relative luminance 0.16, so the warm arm stays red-orange — amber reads as brown and must not be re-attempted. If a light theme ever returns, it is a new decision: reinstate per-theme palettes and the guard's per-mode assertions per ADR-0014, don't auto-derive.
