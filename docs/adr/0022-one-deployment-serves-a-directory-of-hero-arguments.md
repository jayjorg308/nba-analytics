# One deployment serves a directory of hero arguments

The app is **one deployment carrying every hero**: each hero page is a complete argument at its own URL (`/<slug>`), the site root is the **hero index** — a directory of poster tiles, each reusing its hero's banner photo and thesis at tile scale — and navigation between them is plain full-page links (an `<a>` to a URL; no router dependency, no client-side transitions). The **hero registry** (`src/heroes/`) is the single source of hero truth: one config module per hero (thesis, verdict, banner — the ADR-0017/0021 authored copy) with its **colocated verdict guard** (`<slug>.guard.test.ts`), and an ordered registry (`registry.ts`) that the index page, the path router, the sync script, and the per-hero tests all read. Deep URLs resolve via a static-host SPA fallback (`vercel.json` rewrite to `index.html`); the registry modules are node-safe on purpose (no `import.meta.env` — app-layer URL derivation lives in `src/heroes/urls.ts`) so tooling imports the same truth the app renders. This supersedes the branch-per-hero deployment model (the `hero/keyonte-george` branch + Vercel branch alias that validated the player-agnostic engine).

**Why:** the v1 model made a second hero a *parallel deployment* — faithful to "one deployment, one argument," but it walls heroes off from each other (a reader on one hero's page has no path to another's) and every hero costs a standing never-merging branch that drifts from main. The reading of ADR-0018 that actually matters is **no switcher**: the argument structure dies when heroes become view-state under one page — a dropdown re-renders the same page with different numbers, while a directory links *complete pages*. Real URLs with full page loads keep each page a self-contained argument (shareable, its own `document.title`, its own poster), cost zero dependencies, and let static hosting do the routing work. Tiles read straight off the registry so the index can never disagree with what's deployed — registering a hero *is* publishing its tile.

**Consequences:** adding a hero = a config module + colocated guard + web-sized banner photo (ADR-0021) + `npm run hero:sync` (argless mode syncs every registered hero) + a registry entry; the index tile falls out for free. The root URL is now the directory, not the launch hero — hero pages are the shareable arguments. The `hero/keyonte-george` branch becomes reference material, not a deployment target; re-adding George is a registry entry away (his derived data is retained). Do not add a switcher/dropdown to any page — cross-hero navigation is the index tile and the hero page's quiet "All players" footer link, both plain anchors. ADR-0010's consequence "swapping heroes means editing `src/heroConfig.ts`" now reads "adding a hero's module under `src/heroes/`"; the drift guard (HeroPage test) and verdict guard now key off each hero's `slug`/`season` rather than a single config's `payloadUrl`.
