# Matchup is derived per row from a static team map, and unknowns fail the derive

Schema v3 adds per-shot `opponent` (team abbreviation) and `home` so the Shots-view tooltip can carry the game's matchup ("@ PHX" away, "vs OKC" home). The raw row carries the game's `HTM`/`VTM` abbreviations plus the shooter's `TEAM_NAME` (full name); the derive step maps `TEAM_NAME` to its abbreviation through a static 30-team table (`TEAM_ABBREV` in `ingestion/derive_payload.py`, stats.nba.com naming — note "LA Clippers"), then `home = (abbrev == HTM)` and the opponent is the other side. The lookup is per-row, not per-snapshot, so a mid-season trade — `TEAM_NAME` changing partway through a season's rows — stays correct.

**Why:** the UI must not compute this (ADR-0011 — presentation formats, never computes), so the derive seam is where the matchup resolves. A static map beats the alternatives: carrying raw `HTM`/`VTM`/`TEAM_NAME` into the payload just moves the join into every consumer, and resolving names through another endpoint adds a network dependency to a step that must be deterministic (the golden fixture depends on byte-identical output). Per the ADR-0019 ethos, the failure mode is loud, never silent: an unknown `TEAM_NAME` (expansion, rebrand, endpoint drift), or a mapped abbreviation that is neither `HTM` nor `VTM` (map drift), fails the whole derive — a matchup is never guessed into the payload.

**Consequences:** the map is a known maintenance point — an expansion team or rebrand breaks the derive (locally, where pulls run; never in CI or the deployed app) until `TEAM_ABBREV` is updated, which is the intended fix. `opponent`/`home` are descriptive context only: the aggregation reads neither, so the v2→v3 bump changed no metric and no guard assertion, and the Zod schema, derive step, regenerated golden, and re-synced deployed payloads moved together per the cross-language convention (`tests/fixtures/README.md`).
